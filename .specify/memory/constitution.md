# Fitmass Site Constitution

## Core Principles

### I. Server Components First

Every new component MUST default to a React Server Component (RSC).
`'use client'` is permitted only when the component requires browser APIs,
React state (`useState`), or event handlers (`onClick`, `onChange`).
Client components MUST be kept as leaf nodes — never wrap large subtrees
in `'use client'` to avoid unnecessary client bundles.
`page.tsx` files MUST remain thin Server Components: only metadata exports
and top-level data fetching. All UI content MUST live in sibling files.

### II. Auth-Gated Server Actions

Every Server Action in `app/actions/` MUST call `requireSession()`
(EDITOR+) or `requireAdmin()` (ADMIN only) as its first statement.
`redirect()` MUST NOT be wrapped in try/catch — it throws internally.
The `{ error?: string }` return value is reserved for validation failures only.
`revalidatePath()` MUST be called after every successful mutation.
Public-facing routes MUST use `authMode: 'apiKey'`; admin actions MUST
use the default `userPool` auth via `generateServerClientUsingCookies<Schema>`.

### III. Amplify Data Layer (Non-Negotiable)

All data reads and writes MUST go through the Amplify Data client
(`generateServerClientUsingCookies<Schema>` from `@aws-amplify/adapter-nextjs/data`).
Direct DynamoDB or AppSync HTTP calls are prohibited.
Paginated list queries MUST use `listAll()` from `lib/list-all.ts` —
never assume a single page is the full dataset.
The `amplify_outputs.json` file MUST exist at the repo root before the app
will start; it is generated by `npx ampx sandbox` (dev) or
`npx ampx generate outputs` (deployed env).

### IV. Design Token Compliance

All UI styling MUST reference the CSS custom properties declared in
`app/globals.css`. Hardcoding brand colors or font names inline is prohibited.

| Token | Value | Use |
|---|---|---|
| `--color-accent` | `#88BD23` | Brand green — CTAs, highlights |
| `--color-secondary` | `#25B6EB` | Secondary blue — accents |
| `--color-contrast` | `#333333` | Dark backgrounds |
| `--color-surface` | `#F8F8F8` | Light section backgrounds |
| `font-title` | AeroMatics | Headings |
| `font-body` | Encode Sans | Body copy |

Tailwind v4 has no `tailwind.config.js`. Tokens are consumed via
`var(--color-accent)` or Tailwind utility classes when mapped.

### V. Security by Default



User-generated HTML (blog post content) MUST be sanitised with `DOMPurify`
before being passed to `dangerouslySetInnerHTML` — no exceptions.
Image uploads (`POST /api/upload`) MUST validate auth, MIME type, and the
5 MB size limit before writing to S3 or the local filesystem fallback.
Security response headers (`X-Frame-Options`, `HSTS`, `X-Content-Type-Options`,
`Referrer-Policy`, `Permissions-Policy`) are applied globally in `next.config.ts`
and MUST NOT be removed or weakened.
There is intentionally no `middleware.ts` — auth enforcement is done in
`app/admin/layout.tsx` and per-action guards. This boundary MUST be maintained.

### VI. CI Pipeline Compliance (Non-Negotiable)

@docs/pipeline-code-review-automatizado.md

Before finalizing any feature implementation (before opening a PR), the automated
code review pipeline defined in `docs/pipeline-code-review-automatizado.md` MUST
be executed in full. This pipeline is a mandatory quality gate — not optional.

Key obligations:
- Discovery (Etapa 0) MUST be completed and confirmed before any CI file is written.
- Each phase (Fases 1–6) MUST be committed separately with Conventional Commit messages.
- A single PR groups all phase commits — never one PR per phase.
- Gates marked `enforce` in `.github/code-review-pipeline.yml` block merge.
- Gates marked `report` are informative; promotion to `enforce` is a manual team decision.
- No workflow in `.github/workflows/` may commit to the repository automatically.
- Every workflow MUST declare an explicit `permissions:` block with minimum required scope.

## Stack & Environment

**Framework**: Next.js 16 (App Router) · React 19 · TypeScript (strict mode)
**Styling**: Tailwind v4 — tokens in `app/globals.css`; no config file
**Backend**: AWS Amplify Gen 2 — Cognito (auth) · AppSync/DynamoDB (data) · S3 (storage)
**Rich text**: Tiptap v3 (StarterKit + Image + Link) — HTML stored as `Text` in DynamoDB
**Charts**: Recharts (`BarChart`, `PieChart`) — admin dashboard only
**Path alias**: `@/*` resolves to the repository root

**Rendering strategy**:
- `/blog` → ISR, `revalidate: 60`
- `/blog/[slug]` → ISR, `revalidate: 3600`
- Admin pages → Dynamic (session-gated, no caching)

**Auth roles**: `ADMIN` (full access + user management) · `EDITOR` (post CRUD only)
Roles are extracted from `cognito:groups` in the access token via `lib/session.ts`.

**Required env vars**:
- `AMPLIFY_USERPOOL_ID` + `AWS_REGION` — Cognito Admin SDK (user management)
- `AWS_S3_BUCKET` + `AWS_REGION` — S3 uploads (optional; falls back to `/public/uploads/`)
- `NEXT_PUBLIC_GTM_ID` — Google Tag Manager (optional)

## Development Workflow

There is **no test suite**. Quality gates are:
1. `npm run lint` (ESLint) — MUST pass with zero errors before any PR
2. `npm run build` — MUST succeed before merging
3. Manual verification of the golden path and edge cases for any changed feature

**Development commands**:
```bash
npm run dev                  # dev server on localhost:3000
npm run build                # production build
npm run lint                 # ESLint project-wide
npx ampx sandbox             # spin up Amplify sandbox + generate amplify_outputs.json
npx ampx generate outputs    # regenerate outputs from deployed env
```

New features MUST follow the spec-kit workflow:
`/speckit-specify` → `/speckit-plan` → `/speckit-tasks` → `/speckit-implement`

Before opening a PR, the CI pipeline from `docs/pipeline-code-review-automatizado.md`
MUST be executed (Principle VI). This is a hard gate — not a suggestion.

Slug generation uses `lib/slug.ts` (NFD normalisation, diacritics stripped, lowercase).
Collision uniqueness is enforced in app logic — not a DB constraint.

## Governance

This constitution supersedes all other informal practices.
The authoritative runtime reference for tooling, architecture details, and
implementation patterns is `CLAUDE.md` at the repository root.

**Amendment procedure**:
1. Propose the change with a rationale (why the existing principle is insufficient).
2. Update this file and bump the version following semantic versioning:
   - MAJOR: principle removal or redefinition breaking existing code patterns.
   - MINOR: new principle or materially expanded guidance.
   - PATCH: clarifications, wording, typo fixes.
3. Update `CLAUDE.md` if the amendment affects runtime guidance.
4. Commit with message: `docs: amend constitution to vX.Y.Z (<summary>)`

All PRs touching `app/actions/`, `amplify/`, `lib/auth-utils.ts`, or
`lib/session.ts` MUST be reviewed for compliance with Principles II and V.
