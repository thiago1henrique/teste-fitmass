Variável Origem Descrição
AMPLIFY_USERPOOL_ID Amplify Console → Auth ID do Cognito User Pool (ex: us-east-1_Abc123). Usado em app/actions/users.ts e app/admin/team.

Passos para o primeiro deploy

O que o time de implantação precisa fazer

1

Instalar o Amplify CLI

Necessário para criar o backend na sua conta AWS.
npm install -g @aws-amplify/cli

2

Conectar o repositório ao Amplify Hosting

No AWS Console → Amplify → "New App" → "Host web app" → conectar ao GitHub. O arquivo amplify.yml já está configurado no repositório.

3

Provisionar o backend (primeiro deploy)

O Amplify detectará a pasta amplify/ e irá criar automaticamente: Cognito User Pool, tabela DynamoDB via AppSync, e bucket S3.

4

Substituir o amplify_outputs.json

Após o primeiro deploy, baixar o amplify_outputs.json gerado do Console e substituir o placeholder no repositório. Fazer commit e push.

5

Configurar variáveis de ambiente

No Amplify Console → App Settings → Environment variables: adicionar AMPLIFY_USERPOOL_ID, AWS_REGION e AWS_S3_BUCKET com os valores reais.

6

Criar o primeiro usuário admin

No Cognito Console → User Pool → Usuários → "Create user". Em seguida, adicionar ao grupo ADMIN. Ou usar a tela de equipe do admin depois do login.

Pontos de atenção

Decisões de design e tradeoffs conhecidos

⚠️

authorName desnormalizado no Post

DynamoDB não suporta JOINs. O campo authorName é copiado do perfil Cognito no momento da criação do post. Se o usuário alterar seu nome no Cognito, posts antigos continuarão exibindo o nome anterior.

⚠️

Ordenação e filtros feitos em JS (client-side do servidor)

DynamoDB não tem ORDER BY ou filtros complexos nativos no AppSync. Todos os posts são buscados e ordenados em memória (Node.js). Para volumes grandes (> 1.000 posts), considerar usar DynamoDB Streams + Elasticsearch, ou índices GSI específicos.

ℹ️

apiKey permite update em Posts (necessário para view counter)

O schema Amplify Data tem allow.publicApiKey().to(['read', 'update']) para permitir o incremento de views sem autenticação. A apiKey fica apenas no servidor (não exposta ao browser). Pode ser trocado por uma Lambda Function com IAM auth para maior segurança em produção.

ℹ️

View counter: buffer removido (incompatível com serverless)

O padrão de buffer em memória Node.js não funciona em Lambda (cada requisição pode ser um processo diferente). O contador agora é incrementado diretamente a cada visualização. Para alto volume, considerar Amazon Kinesis ou DynamoDB Streams.

✅

Upload de imagens: compatível com ambas as abordagens

O app/api/upload/route.ts verifica a variável AWS_S3_BUCKET: se presente, usa S3. Se ausente, salva em /public/uploads/ localmente. Isso permite desenvolvimento local sem credenciais AWS.

🗄️

Modelo de dados (Post)

Definido em amplify/data/resource.ts

// amplify/data/resource.ts — modelo Post no DynamoDB via AppSync Post: a.model({ id: auto-gerado // UUID (PK do DynamoDB) title: a.string().required(), slug: a.string().required(), summary: a.string().required(), content: a.string().required(), // HTML do Tiptap coverUrl: a.string(), status: a.enum(['DRAFT', 'PUBLISHED']), publishedAt: a.datetime(), views: a.integer().default(0), categories: a.string().array(), authorId: a.string().required(), // sub do Cognito authorName: a.string().required(), // desnormalizado (sem JOIN) createdAt: auto-gerado, updatedAt: auto-gerado, }) .authorization(allow => [ allow.groups(['ADMIN', 'EDITOR']), // escrita completa allow.guest().to(['read']), // blog público (sem auth) allow.publicApiKey().to(['read', 'update']), // view counter ])
