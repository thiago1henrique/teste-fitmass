Importante — esta é uma migração de plataforma

O Amplify não usa PostgreSQL. O banco nativo é DynamoDB (NoSQL), gerenciado pelo Amplify Data com GraphQL via AppSync. A autenticação sai do JWT customizado e passa para o Amazon Cognito. Isso exige reescrita da camada de dados e auth — não é só trocar uma variável de ambiente. O time deve avaliar a disponibilidade antes de iniciar as fases 2 e 3.

Dev responsável

1.2 — Criar estrutura de diretórios
# Estrutura que será criada manualmente amplify/ backend.ts # combina auth + data + storage auth/ resource.ts # Cognito config data/ resource.ts # DynamoDB schema storage/ resource.ts # S3 config
1.3 — amplify/backend.ts

Fase 2 — Autenticação: JWT customizado → Amazon Cognito

Substituir jose + bcryptjs + cookie manual pelo Cognito gerenciado pelo Amplify Auth

Dev responsável
⚠️ Impacto maior: o sistema de login atual (email + senha + cookie JWT) será substituído pelo Cognito. Os usuários existentes precisarão ser migrados ou recriados no Cognito.
2.1 — amplify/auth/resource.ts
import { defineAuth } from '@aws-amplify/backend' export const auth = defineAuth({ loginWith: { email: true, // login por e-mail + senha }, groups: ['ADMIN', 'EDITOR'], // equivalente ao Role do Prisma })
2.2 — Reescrever lib/session.ts
// lib/session.ts — versão Amplify import { getCurrentUser, fetchAuthSession } from 'aws-amplify/auth' import { runWithAmplifyServerContext } from 'aws-amplify/adapter-core' import { cookies } from 'next/headers' import { createServerRunner } from '@aws-amplify/adapter-nextjs' import config from '@/amplifyconfiguration.json' export const { runWithAmplifyServerContext: serverContext } = createServerRunner({ config }) export async function getSession() { return serverContext({ nextServerContext: { cookies }, operation: async (ctx) => { try { const user = await getCurrentUser(ctx) const session = await fetchAuthSession(ctx) const groups = session.tokens?.accessToken?.payload['cognito:groups'] as string[] return { userId: user.userId, email: user.signInDetails?.loginId, role: groups?.[0] } } catch { return null } } }) }
2.3 — Reescrever middleware.ts
// middleware.ts — verifica token Cognito em vez de JWT customizado import { fetchAuthSession } from 'aws-amplify/auth/server' import { NextRequest, NextResponse } from 'next/server' import { createServerRunner } from '@aws-amplify/adapter-nextjs' import config from './amplifyconfiguration.json' const { runWithAmplifyServerContext } = createServerRunner({ config }) export async function middleware(req: NextRequest) { const headers = new Headers(req.headers) headers.set('x-pathname', req.nextUrl.pathname) if (req.nextUrl.pathname.startsWith('/admin') && req.nextUrl.pathname !== '/admin/login') { const authenticated = await runWithAmplifyServerContext({ nextServerContext: { request: req, response: NextResponse.next() }, operation: async (ctx) => { try { const session = await fetchAuthSession(ctx) return !!session.tokens } catch { return false } } }) if (!authenticated) return NextResponse.redirect(new URL('/admin/login', req.url)) } return NextResponse.next({ request: { headers } }) }
2.4 — Reescrever app/actions/auth.ts
// Cognito signIn substitui bcrypt + prisma.user.findUnique import { signIn, signOut } from 'aws-amplify/auth' export async function loginAction(_prev: LoginState, formData: FormData) { try { await signIn({ username: formData.get('email') as string, password: formData.get('password') as string, }) redirect('/admin') } catch { return { error: 'E-mail ou senha inválidos.' } } }

Fase 3 — Banco de dados: PostgreSQL/Prisma → DynamoDB via Amplify Data

Maior esforço da migração. Reescrita de toda a camada de acesso a dados.

⚠️ Maior esforço
⚠️ DynamoDB é NoSQL — não há SQL, JOINs ou colunas. O Amplify Data cria uma GraphQL API (AppSync) na frente do DynamoDB que simplifica o acesso, mas a lógica de consulta muda.
3.1 — amplify/data/resource.ts (schema equivalente ao Prisma)
import { type ClientSchema, a, defineData } from '@aws-amplify/backend' const schema = a.schema({ Post: a.model({ title: a.string().required(), slug: a.string().required(), summary: a.string().required(), content: a.string().required(), coverUrl: a.string(), status: a.enum(['DRAFT', 'PUBLISHED']), publishedAt: a.datetime(), views: a.integer().default(0), categories: a.string().array(), authorId: a.string().required(), }).authorization(allow => [ allow.groups(['ADMIN', 'EDITOR']), allow.guest().to(['read']), // blog público ]), // User gerenciado pelo Cognito — não precisa de model aqui }) export type Schema = typeof schema export const data = defineData({ schema })
3.2 — Reescrever app/actions/posts.ts com Amplify Data client
import { generateServerClientUsingCookies } from '@aws-amplify/adapter-nextjs/data' import { cookies } from 'next/headers' import config from '@/amplifyconfiguration.json' import type { Schema } from '@/amplify/data/resource' const client = generateServerClientUsingCookies<Schema>({ config, cookies }) // Antes: await prisma.post.create({ data: { ... } }) // Depois: await client.models.Post.create({ title, slug, summary, content, status, ... }) // Antes: await prisma.post.findUnique({ where: { slug } }) // Depois: const { data: posts } = await client.models.Post.list({ filter: { slug: { eq: slug } } }) // Antes: await prisma.post.delete({ where: { id } }) // Depois: await client.models.Post.delete({ id })
3.3 — Remover Prisma do projeto
npm uninstall prisma @prisma/client @prisma/adapter-pg pg @types/pg # Deletar: prisma/ lib/prisma.ts

Plano de Rollback

  1. O Amplify Hosting mantém a versão anterior automaticamente se o build falhar — nenhuma ação necessária
  2. Para rollback de código: reverter o commit no GitHub — Amplify faz redeploy da versão anterior automaticamente
  3. Para rollback de DNS: reverter os registros para o servidor anterior — a URL *.amplifyapp.com continua ativa para debugging
  4. Os dados no DynamoDB não são afetados por rollback de hosting ou DNS
